### AC-2 — Delete em cascata de transações ao apagar fatura (TD-06)

**Dado que** existe uma fatura com ID `INV-X` no banco com 5 transações associadas (`invoice_id = INV-X`),

**Quando** o usuário deletar a fatura `INV-X` pela tela Invoices (ou via `dataService.deleteInvoice()`),

**Então:**
- [x] A fatura `INV-X` é removida da tabela `invoices`
- [x] As 5 transações com `invoice_id = INV-X` são removidas da tabela `transactions`
- [x] Após o delete, uma query `SELECT * FROM transactions WHERE invoice_id = 'INV-X'` retorna 0 registros
- [x] Transações de outras faturas NÃO são afetadas
- [x] Transações manuais (`invoice_id IS NULL`) NÃO são afetadas

**Implementação aceita (qualquer uma das duas):**

**Opção A — Constraint de banco (preferida):**
```sql
-- Executar no Supabase SQL Editor
ALTER TABLE transactions
  DROP CONSTRAINT IF EXISTS transactions_invoice_id_fkey;

ALTER TABLE transactions
  ADD CONSTRAINT transactions_invoice_id_fkey
    FOREIGN KEY (invoice_id)
    REFERENCES invoices(id)
    ON DELETE CASCADE;
```
Após executar: a constraint deve aparecer no Supabase Dashboard > Table Editor > transactions > Foreign Keys.

**Opção B — Delete explícito no código:**
```typescript
// dataService.ts — deleteInvoice()
async deleteInvoice(invoiceId: string, userId: string) {
  // 1. Primeiro apaga transações associadas
  await supabase
    .from('transactions')
    .delete()
    .eq('invoice_id', invoiceId)
    .eq('user_id', userId);

  // 2. Depois apaga a fatura
  const { error } = await supabase
    .from('invoices')
    .delete()
    .eq('id', invoiceId)
    .eq('user_id', userId);

  if (error) throw error;
}
```

**Verificação manual:**
1. Importar um PDF de teste com pelo menos 3 transações
2. Ir para /invoices e deletar a fatura importada
3. Verificar no Dashboard que as transações desapareceram
4. Verificar no Supabase Table Editor que não há registros órfãos

### AC-3 — `saveTransactions()` usa UPSERT (TD-07)

**Dado que** o usuário importa a mesma fatura PDF duas vezes (por acidente),

**Quando** `dataService.saveTransactions()` for chamado na segunda importação,

**Então:**
- [x] As transações NÃO são duplicadas no banco de dados
- [x] O número total de transações referentes àquela fatura é o mesmo antes e depois da segunda importação
- [x] Nenhum erro é lançado (o UPSERT deve ser idempotente)

**Implementação exigida:**

O `upsert` deve usar uma constraint única definida no banco. A chave natural para idempotência é a combinação:
```
user_id + purchase_date + description + amount
```

**Passo 1 — Adicionar constraint UNIQUE no Supabase:**
```sql
-- Executar no Supabase SQL Editor
-- NOTA: usar IF NOT EXISTS para ser re-executável com segurança
ALTER TABLE transactions
  ADD CONSTRAINT IF NOT EXISTS transactions_natural_key_unique
  UNIQUE (user_id, purchase_date, description, amount);
```

**Passo 2 — Atualizar dataService.ts:**
```typescript
// ANTES (linha 57)
const { error } = await supabase.from('transactions').insert(txsToSave);

// DEPOIS
const { error } = await supabase
  .from('transactions')
  .upsert(txsToSave, {
    onConflict: 'user_id,purchase_date,description,amount',
    ignoreDuplicates: true
  });
```

**Verificação:**
```bash
# Testar importando o mesmo PDF duas vezes
# Contar registros antes da segunda importação:
# SELECT COUNT(*) FROM transactions WHERE invoice_id = '<id>';
# Contar registros após a segunda importação — deve ser o mesmo número
```

**Atenção:** Se a constraint `UNIQUE` não existir no banco antes de usar `onConflict`, o Supabase lançará erro. O passo do banco DEVE ser executado antes do deploy do código.

  const map: Record<string, string> = {};

    if (f.id) {  // Versão harmonizada — sempre verifica f.id

**Assinatura exata esperada de `getCycleInfo`:**
```typescript
export function getCycleInfo(
  t: Transaction,
  invoiceDateMap: Record<string, string>
): { label: string; order: number } {
  let refDateStr = '';
  if (t.invoiceId && invoiceDateMap[t.invoiceId]) {
    refDateStr = invoiceDateMap[t.invoiceId];
  } else {
    refDateStr = t.purchaseDate || t.date;
  }
  if (!refDateStr) return { label: 'Sem Ciclo', order: 0 };
  const refDate = new Date(refDateStr);
  const month = refDate.getUTCMonth();
  const year = refDate.getUTCFullYear();
  const monthNames = ['Jan', 'Fev', 'Mar', 'Abr', 'Mai', 'Jun', 'Jul', 'Ago', 'Set', 'Out', 'Nov', 'Dez'];
  return {
    label: `${monthNames[month]}/${year.toString().slice(-2)}`,
    order: year * 100 + month
  };
}
```

## Riscos

| Risco | Probabilidade | Mitigação |
|-------|--------------|-----------|
| UNIQUE constraint falha em dados existentes duplicados | Média | Antes de criar a constraint, rodar `SELECT user_id, purchase_date, description, amount, COUNT(*) FROM transactions GROUP BY 1,2,3,4 HAVING COUNT(*) > 1` para identificar duplicatas e limpá-las |
| CASCADE delete remover transações indesejadas | Baixa | Testar em staging primeiro com fatura de teste. O CASCADE só afeta `invoice_id NOT NULL` |
| Edge Function não deployada break extração em produção | Alta se esquecida | Verificar `supabase functions list` ANTES de ativar `VITE_USE_EDGE_FUNCTION=true` em prod |
| Dashboard ou Reports quebrar após migração de cycleUtils | Baixa | Verificar dados visuais após cada Task 2.x antes de prosseguir |
