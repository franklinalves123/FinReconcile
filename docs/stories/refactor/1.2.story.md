# Story 1.2 — Módulo de IA: Isolamento e Categorização Real com Gemini (Fase 3)

## Descrição

Como desenvolvedor do sistema FinReconcile, preciso isolar toda a lógica de Inteligência Artificial em um módulo dedicado `services/ai/` e implementar a função `categorizeTransactions()` com chamada real ao Gemini — passando as categorias do usuário como contexto — para que o sistema deixe de retornar sempre `'Outros'` e passe a sugerir categorias precisas automaticamente após a importação de cada fatura.

> **Contexto Jarvis:** Esta é a história mais estratégica do ciclo de refatoração. O módulo `services/ai/` é a fundação do cérebro do Jarvis — o ponto de entrada para toda inteligência futura do sistema (categorização, insights, detecção de anomalias, processamento de linguagem natural). Ao isolar a IA aqui, habilitamos: (1) troca de provider sem tocar nos componentes React; (2) evolução independente de extração vs. categorização vs. insights; (3) testes unitários isolados. **Esta story resolve TD-04 e é pré-requisito para a Fase 4** (decomposição do App.tsx).

  Retorne um objeto JSON com array "suggestions".
  Mantenha a mesma ordem das descrições.
  Use APENAS as categorias da lista. Se não souber, use 'Outros' com confidence 0.3.

  ${descriptions.map((d, i) => `${i + 1}. "${d}"`).join('\n')}

### AC-6 — `services/ai/categorizer.ts`: implementação REAL com Gemini (resolve TD-04)

**Dado que** `categorizeTransactions()` em `geminiService.ts:109-113` é um stub que sempre retorna `'Outros'`,

**Quando** o desenvolvedor criar `services/ai/categorizer.ts`,

**Então** a função exportada deve ter a seguinte assinatura exata:
```typescript
export async function categorizeTransactions(
  descriptions: string[],
  availableCategories: string[]
): Promise<CategorySuggestion[]>
```

**A implementação DEVE:**

- [ ] **Retornar um array de fallback** quando `descriptions` for vazio ou `availableCategories` for vazio, sem chamar o Gemini:
  ```typescript
  if (descriptions.length === 0 || availableCategories.length === 0) {
    return descriptions.map(d => ({
      description: d,
      suggestedCategory: 'Outros',
      confidence: 0
    }));
  }
  ```

- [ ] **Fazer UMA ÚNICA chamada** ao Gemini para todas as descrições (não uma chamada por transação)

- [ ] **Usar `ai.models.generateContent()`** importado de `./client.ts`

- [ ] **Usar `buildCategorizePrompt(descriptions, availableCategories)`** de `./prompts.ts`

- [ ] **Usar `responseSchema` estruturado** para garantir output válido:
  ```typescript
  config: {
    responseMimeType: "application/json",
    responseSchema: {
      type: Type.OBJECT,
      required: ["suggestions"],
      properties: {
        suggestions: {
          type: Type.ARRAY,
          items: {
            type: Type.OBJECT,
            required: ["description", "suggestedCategory", "confidence"],
            properties: {
              description:          { type: Type.STRING },
              suggestedCategory:    { type: Type.STRING },
              suggestedSubcategory: { type: Type.STRING },
              confidence:           { type: Type.NUMBER }
            }
          }
        }
      }
    }
  }
  ```

- [ ] **Garantir que o resultado tem o mesmo tamanho que o input.** Se o Gemini retornar um número diferente de sugestões, preencher as faltantes com fallback `{ description, suggestedCategory: 'Outros', confidence: 0 }`

- [ ] **Tratar erros com fallback total:** se a chamada ao Gemini falhar por qualquer motivo (rede, quota, parsing), retornar array de fallback para TODAS as descrições (não lançar exception):
  ```typescript
  } catch (error) {
    console.error("Erro na categorização por IA, usando fallback 'Outros':", error);
    return descriptions.map(d => ({
      description: d,
      suggestedCategory: 'Outros',
      confidence: 0
    }));
  }
  ```

- [ ] **O modelo usado** é `'gemini-1.5-flash'` (lido de `aiConfig.model`)

- [ ] **NÃO chamar via Edge Function** nesta story — usar sempre chamada direta (Layer 1 é suficiente para texto)

**Verificação funcional (obrigatória):**
- [ ] Após implementar, importar um PDF com pelo menos 5 transações diversas
- [ ] **Pelo menos 3 de 5 transações** devem receber categoria diferente de `'Outros'`
- [ ] Transações com descrições óbvias devem ser categorizadas corretamente:
  - Qualquer descrição contendo `UBER`, `99APP`, `TAXI` → deve sugerir `'Transporte'`
  - Qualquer descrição contendo `IFOOD`, `RAPPI`, `RESTAURANTE`, `PADARIA` → deve sugerir `'Alimentação'`
  - Qualquer descrição contendo `FARMACIA`, `DROGASIL`, `DROGA` → deve sugerir `'Saúde'`
  - Qualquer descrição contendo `NETFLIX`, `SPOTIFY`, `AMAZON` → deve sugerir `'Serviços'`

### Mudança de tipo de retorno de `categorizeTransactions`

**Antes:** `Record<string, string>` — mapa de `description → categoryName`

**Depois:** `CategorySuggestion[]` — array com objeto rico por descrição

**Impacto:** Somente `App.tsx:119-132` usa esse retorno. Nenhum outro consumidor existe no codebase.

**Acesso ao dado:**
```typescript
// ANTES

// DEPOIS
const suggestion = aiSuggestions.find(s => s.description === item.description);
const category = suggestion?.suggestedCategory || 'Outros';
const subcategory = suggestion?.suggestedSubcategory;  // novo campo!
const confidence = suggestion?.confidence;              // novo campo!
```

### Design do Prompt de Categorização — Diretrizes

O prompt em `buildCategorizePrompt` deve:
1. Ser em português (BR) — as descrições de fatura são em PT-BR/inglês abreviado
2. Incluir explicitamente as categorias disponíveis como lista separada por vírgula
3. Pedir `suggestedSubcategory` apenas quando for uma subcategoria óbvia das categorias do sistema (ex: `'Restaurante'` quando a categoria é `'Alimentação'`)
4. Instruir o modelo a usar somente categorias da lista (nunca inventar novas)
5. Ser conciso — o Gemini 1.5 Flash é rápido com prompts curtos

### Garantia de correspondência description → suggestion

O `find(s => s.description === item.description)` em `App.tsx` depende de o Gemini retornar o campo `description` exatamente como foi enviado no input. O `responseSchema` força o tipo mas não garante o valor exato. Para robustez, o fallback `|| 'Outros'` em `App.tsx` já trata os casos onde `find` retorna `undefined`.

### Estratégia de batching (custo de tokens)

A implementação usa **uma única chamada Gemini por fatura** (não uma por transação). Uma fatura típica tem 10-40 transações. Uma chamada com 40 descrições custa aproximadamente 200-500 tokens de input — negígivel para `gemini-1.5-flash`. Isso evita problemas de rate limit e custo linear.

### Layer 2 para categorizer — fora do escopo desta story

O `extractInvoiceData` tem suporte a Layer 2 (Edge Function) porque envia um PDF como base64 — payload grande, chave exposta no bundle. O `categorizeTransactions` envia apenas texto puro — payload pequeno. Por consistência de segurança, Layer 2 pode ser adicionado depois, mas não é obrigatório nesta story. O `client.ts` foi projetado para facilitar essa extensão futura.

## Riscos

| Risco | Probabilidade | Mitigação |
|-------|--------------|-----------|
| Gemini retornar `description` diferente da string enviada | Média | `find()` com fallback `|| 'Outros'` já está no AC-9. Mesmo que não encontre, a transação recebe 'Outros' silenciosamente |
| Gemini retornar array com tamanho diferente de `descriptions.length` | Baixa | Verificação explícita de tamanho no `categorizer.ts` + preenchimento com fallback (AC-6) |
| Custo de tokens elevado | Muito baixa | Uma chamada por fatura. Gemini 1.5 Flash: ~$0.075/1M tokens. Uma fatura com 40 transações = ~500 tokens input ≈ frações de centavo |
| Regressão na extração de PDF | Baixa | Task B.2 exige verificação de regressão ANTES de avançar para Fase C |
| TypeScript errors no `geminiService.ts` deprecated | Baixa | O re-export usa os tipos de `ai/index.ts` — mesmos tipos, sem breaking change |
| Subcategorias inventadas pela IA | Média | O prompt instrui o modelo a usar apenas subcategorias conhecidas. Se a IA inventar, o campo `subcategory` apenas terá um valor não-padrão — não causa crash, apenas inconsistência visual na Review |
